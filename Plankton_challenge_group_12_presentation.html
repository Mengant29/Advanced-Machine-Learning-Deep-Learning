Train/test split¶

Since some categories have few elements, we do not want the training data to have none of them (there is a probability that the testing set to contain all or most of the elements of such categories). If it were to happen it would not be possible to classify whole categories.

On the other way around, if all the elements are in the training set, the F1 accuracy measure would be biased toward an overestimation of success.

To avoid this, we chose to use a Stratified Shuffle Split that ensures that the data will be evenly split between the training, validation and testing sets.

Performance evaluation¶

Defining a measure of performance¶

We not only use the accuracy but also the F1 score that takes into account the false positive/negative predictions. Choosing average=macro favorises the prediction of small classes. It means that considering a plankton as a detritus is more penalizing than considering a detritus as a certain plankton. This is also related to our choice of a weighted categorical crossentropy biased toward small classes.

Conclusion¶

At first we trough that the random forest was better. The cross validation accuracy during the gride search was better for the random forest (0.796) than for the logistic regression (0.780). But when we looked at the f1 macro score it was quite the opposite (respectively 0.270 and 0.350). The f1 macro score of the logistic regression is much greater than the one for the random forest.

Our hypothesis is that the random forest is optimizing the accuracy by favorizing the prediction of bigger classes of the dataset, and forgetting the small ones. This seems not to be the case for the logistic regression.

We can see it clearly when looking at the confusion matrices, the diagonal of the random forest one is brighter but has much more holes compared to the logistic regression one, where the diagonal is more "complete" and less bright.

The main source of confusion is labelling plankton as detritus (first row).

Data generator¶

A bad idea would have been to use our previous functions to ceate the dataset of resized pictures of 100x100 pixels. However, it would have occupied too much space in memory.

We need to load our pictures "on the fly", in a more efficient way: we want real-time data feeding of our Keras model. For the data generator, we got inspired by this website : https://stanford.edu/~shervine/blog/keras-how-to-generate-data-on-the-fly

Since the images shapes are variable, we need a function to preprocess those images into a fixed shape of 100x100 so that we can feed our models. As the background of the images is white, to resize the images, we just fit the rectangular orginal image in a 100x100 square, and then fill the remaing area with white.

We have not found any implementation of such resizing preprocessing, so we implemented the function by ourselves.

In [1]:
def resizeImage(img, w=100, l=100):
    w_, l_ = img.shape
    scale_factor = min(float(w)/float(w_), float(l)/float(l_))
    img = rescale(img, scale_factor, anti_aliasing=True, mode='reflect', multichannel=False)
    Dw, Dl = w - img.shape[0], l - img.shape[1]
    img = np.pad(img, ((int(np.ceil(Dw/2)), int(np.floor(Dw/2))),(int(np.ceil(Dl/2)), int(np.floor(Dl/2)))), mode='constant', constant_values=1.)
    return img

def accessImage(img_id, w=100, l=100): # Allows us to access the "resized" image
    img = np.array(Image.open(img_files['imgs/'+str(img_id)+'.jpg']))
    return resizeImage(img, w, l).reshape(w, l, 1)

class DataGenerator(Sequence):
    'Generates data for Keras'
    def __init__(self, ids, labels, batch_size=32, dim=(100,100), n_channels=1,
                 n_classes=num_classes, shuffle=False, data_augmentation=False):
        'Initialization'
        self.dim = dim
        self.batch_size = batch_size
        self.labels = labels[:]
        self.list_IDs = ids[:]
        self.n_channels = n_channels
        self.n_classes = n_classes
        self.shuffle = shuffle
        self.on_epoch_end()
        self.data_augmentation = data_augmentation
        if self.data_augmentation:
            self.datagen = ImageDataGenerator(
                rotation_range=180,
                width_shift_range=0.1,
                height_shift_range=0.1,
                horizontal_flip=True,
                vertical_flip=True,
                fill_mode='nearest')

    def __len__(self):
        'Denotes the number of batches per epoch'
        return int(np.floor(len(self.list_IDs) / self.batch_size))
     
    def on_epoch_end(self):
        'Updates indexes after each epoch'
        # Assuming it preserves the correlation between labels and picture ids
        self.indexes = np.arange(len(self.list_IDs))
        if self.shuffle == True:
            np.random.shuffle(self.indexes)

    def __getitem__(self, index):
        'Generate one batch of data'
        # Generate indexes of the batch
        indexes = self.indexes[index*self.batch_size:(index+1)*self.batch_size]

        # Find list of IDs
        list_IDs_temp = [self.list_IDs[k] for k in indexes]

        # Generate data
        X, y = self.__data_generation(indexes, list_IDs_temp)

        return X, y

    def __data_generation(self, indexes, list_IDs_temp):
        'Generates data containing batch_size samples' 
        # Initialization
        # X : (n_samples, *dim, n_channels)
        X = np.empty((len(indexes), *self.dim, self.n_channels))
        y = np.empty((len(indexes),1), dtype=int)
        
        if self.data_augmentation:
            X, y = next(self.datagen.flow(X, y, batch_size=self.batch_size, shuffle=False))
        
        # Generate data
        for i in range(len(indexes)):
            # Store sample
            ID = list_IDs_temp[i]
            X[i,] = accessImage(ID, self.dim[0], self.dim[1])
            # Store class
            y[i] = self.labels[indexes[i]]
        # return X, y
        return X, to_categorical(y, num_classes=self.n_classes)

Initialisation: We put as arguments relevant information about the data, such as dimension sizes, number of channels, number of classes, batch size, or decide whether we want to shuffle our data at generation. We also store important information such as labels and the list of IDs that we wish to generate at each pass.

On epoch end: Here, the method on_epoch_end is triggered once at the very beginning as well as at the end of each epoch. If the shuffle parameter is set to True, we will get a new order of exploration at each pass (or just keep a linear exploration scheme otherwise).

data generation: It produces the batches of data. During data generation, this code reads the NumPy array of each example from its corresponding file ID.npy. Since our code is multicore-friendly, note that you can do more complex operations instead (e.g. computations from source files) without worrying that data generation becomes a bottleneck in the training process. Also, please note that we used Keras' keras.utils.to_categorical function to convert our numerical labels stored in y to a binary form suited for classification.

len: Each call requests a batch index between 0 and the total number of batches, where the latter is specified in the len method. A common practice to set this value is to ensure that the model sees the training samples at most once per epoch.

getitem: to generate the batch corresponding to a given index

Note: If we choose data_augmentation=True, we use an ImageDataGenerator from keras that allows data augmentation. According to what we saw from those images, we enable flipping, shifting and rotation. But we found out that using data augmentation not only slowed down the execution, but produced slightly worse results. It may be caused by the fill_mode set to nearest that could produce artifacts in the augmented images. Or this is maybe due to 45° rotations that zoom out in the images, loosing details. Unfortunately, we have not had enough time to investigate this further.

Defining the loss¶

We first used the already-built categorical_crossentropy, but it doesn't take into account the imbalanceness of the data. Since the planktons are in minority compared to detritus or feces, we do not want our model to only classify images as detritus/feces (that leads to around 65% accuracy). A way to incentive this is to use a weighted categorical crossentropy that penalizes more the wrong labelling of the less represented classes. We want a greater weight for smaller categories, but we do not want a too huge gap between the weights. That is why we use a logarithmic scale to calculate those weights:

$$w_k = \frac{1}{log(c_k) + \lambda}$$

where $c_k$ is the number of elements in class $k$, and $\lambda$ is a parameter we can tune: smaller $\lambda > 0$ leads to greater gaps (there is a max bound for this gap), and bigger $\lambda$ leads to a more uniform weighting.

We used model 1 to tune this parameter and found out that a value very close to zero is better to maximize the F1 score we defined (see the paragraph dedicated to the measure of performance).

Note: Keras has no implementation of weighted categorical cross entropy, so we implemented it by ourselves.

weights = tf.constant(unnomalized_weights/unnomalized_weights.sum())

def weighted_categorical_crossentropy(target, output, axis=-1):
    # scale preds so that the class probas of each sample sum to 1
    output = output / tf.math.reduce_sum(output, axis, True)
    # Compute cross entropy from probabilities.
    epsilon_ = tf.keras.backend.epsilon()
    output = tf.clip_by_value(output, epsilon_, 1. - epsilon_)
    return -tf.reduce_mean(tf.math.reduce_sum(weights * (target * tf.math.log(output) + (1 - target) * tf.math.log(1-output)), axis))

Defining the optimizer¶

We use the already built-in Adam optimizer that has been proven to be quite efficient by the litterature about optimization.

Models¶

Since we are dealing with images, we found it relevant to use CNN models (Convolutional Neural Network).

Model 0¶

For a first model, we got inspired from LeNet5 where a few convolutional layers with an increasing number of channel are alterned with MaxPooling layers to decrease the feature maps dimensionality. The final convolutional output before flattening is of shape (128, 2, 2) and the flattened version is directly connected to the output. To prevent vanishing/exploding gradient issues, we use the relu activation at each layer, except for the last one where we require an output between 0 and 1 for the loss calculation.

Model 1¶

Model 1 is a variation of Model 0 with one less convolutional layer, but with an extra fully-connected layer just after the flattening. The shape of the tensor just before the flattening is (64, 4, 4). We chose greater kernel sizes in general to reduce slightly more the feature maps dimensionality.

backbone models¶

We tried to use pretrained backbone feature extractors from keras.application combined with fully-connected layers to see if they performed better than what we did so far.

We first encountered the problem of the image number of channels (that can be solved by converting our grayscale images to RGB format). Then we found out that the training of the fully-connected layers (by freezing the backbone) is way too slow (1 batch per second, and we have 1421 batches per epoch...). If we had more time, or Zoe executions of 6hrs+, we may have consider to further explore these backbone techniques.

Conclusion¶

Model 1 seems rather good, indeed it achieves the highest F1 macro score (38.8%), just followed by the logistic regression that achieves 35.0%. This is difficult to say if we can possibly much more increase this score since the problem seems to come from the classes that are under-represented, and completely forgotten by all our models.

As the choice of logarithmic scaling for the weighting of the categorical crossentropy is arbitrary, maybe changing its formula could benefit, by increasing the weight of those forgotten classes. But on the other hand, we do not want the distribution of the classification differing too much from the orginal ones (see the pies). This problem is quite challenging and we have not had enough time to further explore it.

Nonetheless, eventhough the accuracy is a bit lower compared to the random forest, the F1 score indicates that Model 1 is our best model so far. Maybe that upgrading this particular model with dropout, proper data augmentation or more complexity, while benefitting from the best possible weighted crossentropy, can overtake our current results. But as we said, we have not had enough time to completely perfect our models.
